## Vad som genomfördes

### Plan-design + ADR via CC (utan webb-Claude)

CC drog initial plan, fick design-validering från `dotnet-architect` med 2 kritiska + 8 viktiga feedbacks innan kod skrevs:

**Kritiska:**
1. AggregateId från response, inte command — `IAuditableCommand<TResponse>` med `ExtractAggregateId(response)`
2. Pipeline-placering — *initialt* tolkades arkitekt-rapporten fel (Audit före UoW), CC korrigerade under implementation till Audit innerst (efter UoW). Atomicitet kräver att Audit:s post-action lägger entity i DbContext och UoW:s post-action SaveChanges sparar allt i samma transaction.

**Viktiga:**
3-10: ICorrelationIdProvider + IRequestContextProvider som nya portar, occurred_at via IDateTimeProvider i behavior, Application/Common/Auditing/-folder, ingen separat IAuditLogger-port (behavior skriver direkt mot IAppDbContext), skip på Result.Failure i Fas 1, ingen payload Fas 1, AuditLogEntry flat entity i Domain/Auditing/.

ADR 0022 skrevs efter feedbacks applicerats.

### Reviews

**code-reviewer:** Approved. 0 Blocker, 0 Major, 6 Minor (alla informationella, defereras till Fas 2/4). Notable: pipeline-ordning-låsning via architecture test rekommenderades men deferreras till Fas 2 när Worker-pipeline aktiveras.

**security-auditor:** 0 Critical, 3 Major, 4 Minor.

Major-fynd som åtgärdades:

1. **IP-PII per Breyer-domen + WP29:** rå IP klassas som personuppgift. Fix: `RequestContextProvider.Anonymize(...)` maskar `/24` IPv4 / `/48` IPv6 — bevarar geo-region för incident-response, eliminerar unique fingerprint.

2. **GDPR Art. 17-policy saknades:** vid kontoradering hanteras inte audit-rader. Fix: ADR 0022-sektion "GDPR-policy" som specificerar:
   - `correlation_id`, `event_type`, `aggregate_type`, `aggregate_id`, `occurred_at` behålls 90 dagar (Art. 5(2) accountability)
   - `user_id`, `ip_address`, `user_agent` anonymiseras till NULL
   - Implementation-cascade tracked som **TD-16** (blocker för Fas 1 prod-deploy).

3. **`X-Correlation-Id` user-controlled audit-spoofing:** angripare kunde injicera Guid och korrelera sin trail med legitim users. Fix: `CorrelationIdProvider` läser INTE längre headern — alltid server-genererat per OWASP ASVS V7.1.4.

Minor #1 (MarkGhosted XML-doc) och #3 (ImpersonatedBy TODO) också fixade.

Efter fixarna: 419/419 tester fortfarande gröna.

## Tekniska beslut

- **Pipeline-behavior framför domain event subscriber.** Domain event dispatcher finns inte i kodbasen idag. Att bygga en är separat ADR + scope. Pipeline-behavior är atomiskt rätt och scope-effektivt.
- **AuditBehavior INNERST i pipelinen.** Audit:s post-action lägger entity i DbContext, UoW:s post-action SaveChanges persisterar atomiskt.
- **Generic `IAuditableCommand<TResponse>` med `ExtractAggregateId`.** Löser Create-fall (response.Value) vs mutation (command-fält) elegant.
- **Skip på Result.Failure i Fas 1.** Failed-attempts-audit retro-fittas i Fas 6.
- **Ingen payload i Fas 1.** PII-sanering deferras till Fas 4 — kolumnen reserverad i schema men null.
- **Ingen FK mot users/job_seekers.** Write-only-disciplin, får inte hindras av soft-delete-cascades.
- **IP-anonymisering /24+/48 framför borttagning.** Behåller geo-region för incident-response — IP-borttagning hade krävt migration-rollback.
- **Server-genererat correlation-ID.** OWASP ASVS V7.1.4. X-Correlation-Id-header läses inte längre.
- **Worker-pipeline-registrering deferras till Fas 2.** Worker-shell är tom (ADR 0010), ingen Mediator/Application-DI ännu.
- **Plan-design via CC + dotnet-architect-validering.** Mönster från STEG 7 fungerar för väl-avgränsad infrastruktur. ADR skrevs efter design-feedback applicerats.
